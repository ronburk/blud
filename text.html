Introduction

In the nearly half-century since Feldman's make utility was described in Software: Practice and Experience, innumerable tools for software configuration management have emerged. Newer tools tend to be complex, often targetting large projects that make up a minority of use cases. Older but still maintained tools such as GNU make that should be more appropriate for the majority of projects that are of modest size have failed to adapt to modern practices, such as the need for multiple builds into multiple output directories.

blud is a simple modernization of the original make utility that focuses on two design goals: the most common tasks should be trivial, and less common tasks should be possible using a simple, widely-known, well-documented programming language: Lua.

Quick Look

Without stopping to explain the syntax, here is a simple blud script that can work for most any beginning-stage C programming project: 

myprog:: *.c
It says that myprog is an executable that should be built by compiling all of the C files in the current directory. The :: operator allows rules to refer to source files rather than object files; this is often easier than the traditional : operator (also supported, of course) that refers to object files.

Unlike the days when make was invented, modern programmers often want to store the products of their software builds in a separate directory from the source code. blud addresses this by adding a new implicit directory to the current working directory that most build utilities implicitly rely on: the output working directory. Here, the previous blud script is revised to store all its automatically generated files in the ./output directory: 

OWD = ./output
myprog :: *.c
OWD is a blud variable that specifies where the products of building your software reside, much like the current working directory implicitly specifies where source files are to be found. As usual with make-style utilities, you could instead (or in addition, to override) specify the value of the OWD variable on the blud command line: blud OWD=./output

Programmers also often desire to regularly build the same project in more than one way. For example, they may want to create a debug build that contains symbol tables to ease debugging, alongside a release build without such symbols to make the executable smaller. 

debug   : OWD = ./debug
debug   : CFLAGS = -g
debug   : myprog
release : OWD = ./release
release : myprog
myprog :: *.c
Given this script, typing blud debug first sets the OWD variable to the appropriate subdirectory, sets an appropriate compiler flag for generating symbolic tables, then (because debug : myprog says that the target named debug depends on the target named myprog) builds myprog. blud release does the same but for the ./release subdirectory and (the default) without symbol tables.

However, because this is such a common case and blud is designed to make common tasks trivial, blud offers a simplified syntax: 

{debug}   : myprog CFLAGS = -g
{release} : myprog
myprog   :: *.c
The {directory-name} syntax for a target creates the directory if it does not exist (a detail not handled by the previous more elaborate script) and sets the output working directory to that location.

blud is a portable build utility that uses Lua as its extension language. It addresses two drawbacks of most make-derived designs (such as GNU make).

First, it adds a second implicit directory to the current working directory that all such utilities rely on. The output working directory makes it simple to allow more than one build, each in its own separate directory (e.g., debug and release builds).

Second, rather than add features by trying to extend the simple macro language of original make, beneath the covers blud is essentially a preprocessor for Lua, a widely-known and well-documented scripting language. This means that when a task arises that is not easily handled by conventional make rules, the user can always switch to writing explicit Lua code to handle most any task imaginable.

This documentation assumes you are familiar with most any build utility that descends from make. It also assumes you either have a passing familiarity with Lua or are willing to go watch a quick tutorial. If you know JavaScript, you can view Lua as a similar but vastly simpler language.

Overview

Although you can write simple make scripts for blud and never actually use any Lua code, if you understand how make directives are integrated with Lua, you will be able to switch to writing Lua whenever it is more convenient or required to do something out of the ordinary.

Consider the following simple blud script: 

program: program.c utility.c
    cc -o program program.c utility.c
The basic idea of any make is to construct a dependency graph from such rules:

To build a file named in the rules, such as program, any make utility first recursively checks all its children in the graph. Each file named in the rules can have actions associated with it that are presumed to update that file. For each child that is newer than its parent, make executes its associated actions to update it. Thus, the basic two phases of a make utility are: build a dependency graph, then traverse it checking file timestamps to execute actions to update that file if necessary.

blud allows you to impose your own custom Lua code into nearly every aspect of these two phases: build a graph and traverse the graph. Your particular build may have no need for customizing blud, but understanding its structure in some detail can make it easier to understand why things work the way they do.

Terminology

In the blud script above, the only rule involves three files: program, program.c, and utility.c. Files on the left side of the colon (program) are called targets, while files appearing on the right side of the colon (program.c and utility.c) are called prerequisites.

The term atom refers generically to something that appears on either the left or right side of a rule. The term is a useful reminder that these are not always files. For example, sometimes it is useful to have an atom that simply serves as a label: 

clean:
    rm program.o utility.o program
Assuming no file named clean actually exists, this rule means that blud clean will simply execute a command that removes the products that may have been created by running blud program. When a rule has no prerequisites, the targets are presumed to need updating, so their actions are executed.

The term atom also helps remind us that the names in a rule are usually not precisely files. For example, the name program.c only uniquely identifies a file when it is combined with the implicit current working directory. Thus there is a step in which blud must uniquely associate the atoms in a rule with files. We call this step binding and after an atom is bound it has a bound name, which is a precise path.

Invoking make-style Code from Lua

blud allows you to mix Lua and make syntax somewhat freely. It uses a fairly simple set of rules to decide whether it is reading Lua code that will be passed through or blud directives that will be translated to Lua before being passed through. In general, the dollar sign ($) inside Lua code signals the preprocessor that what follows should be interpreted in the manner of a make-style script.

First, any line starting with a non-whitespace character is examined to see if it looks like a make-style rule or variable assignment. If it does, it is translated to Lua code. Example: 

-- next line is legal Lua, but will be treated as make-style variable assignment
CFLAGS = -g
-- Simply indent to force the same line to be interpreted as Lua:
    -- assign the opposite of Lua variable "g" to Lua variable "CFLAGS"
    CFLAGS = -g
-- some might prefer a more verbose difference in syntax:
do
    CFLAGS = -g  -- definitely Lua code, not a make-style assignment
end
Second, when a line looks like a make-style rule, blud will translate it to Lua code and attempt to gather any following lines that could be part of a recipe for that rule. These are any indented, non-blank lines that follow target/prerequisite line: 
program : foo.o
    # next line is passed to shell
    foo=7

    echo "blank lines mean end of recipe, so this is erroneous Lua code!"
Third, inside Lua code, any $(...) construct will turn into a string that contains what the corresponding make-style expression would produce: 
do
    if  $(CFLAGS):find("-g") then
        print("-g already part of CFLAGS")
  -- ...
end
Finally, Lua code can also contain complete make-style rules and recipes, by preceding a line with a dollar sign and whitespace: 
if $(OS) == "Windows" then
    -- handle special Windows helper utility
    $ winhelper.exe : winhelper.o
    $     cl $< /o $@
end

Feldman's original make was a marvel of simplicity; a simple syntax for describing a dependency graph along with the actions required to rebuild missing dependents. One can implement a basic version of it in a single page of awk script.

The syntactic challenge for any modern make, however, is the build. Programmers often want to create slightly different builds of the same (or nearly the same) source files (a Windows build, a Linux build, a release build, a debug build, etc.). This is always possible in even the simplest make by just duplicating and then adjusting the original set of dependency graph rules but of course keeping nearly duplicate rule sets in sync is verbose and error-prone.

Feldman's make made liberal use of recursive macros and special character sequences to add features to the basic simple syntax. This trend often continued in make derivatives to the logical conclusion of drifting into incorporating a scripting language made cryptic by its roots. Incrementally developing a language starting from the primitives of make tends to result in a language that satisfies no one.

Instead of bolting on yet another peculiar language to the basic make syntax, blud is built on a simple, efficient, widely-known scripting language: Lua. The basic Lua syntax is extended to include the very simple basic make syntax for specifying dependencies.

Overview

Here is a simple blud script: 

program: program.c
    cc -o program program.c
As you can see it would also be valid input for make, or even most utilities derived from make.

Assuming this script was in a file called blud, executing the blud command would first rewrite this as Lua code into a file named blud.lua and then execute that code. Here roughly what blud.lua might look like in this case: 

-- DO NOT MODIFY: Generated by blud.
table.insert(OBJECTS["program"].PREREQUISITES, "program.c");
table.insert(OBJECTS["program"].ACTIONS, "cc -o program program.c");
BUILD.default = "program"
build()

Now suppose we want to have two different builds for our program: one will include debugging symbols and the other will not. 

debug : debug/program
release : release/program

debug/program : program.c
    cc -g -o debug/program program.c
release/program : program.c
    cc -o release/program program.c
Again, this is a valid specification for make or almost any make derivative. However, the rules have begun to become repetitive and have begun to obscure the fact that the debug and release builds are identical except for the location they store their object files and the compiler flags they use.

This is the fundamental syntactic challenge of any make utility: allowing arbitrary build customizations while succinctly capturing their similarities.

Note that make always contained a concept of implicit directory. There is no mention in the script of where program.c resides; it is assumed to reside in the current directory, that is, the directory the user was in when they ran blud. This idea leads to the first extension blud adds to basic make.
