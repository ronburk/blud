Introduction

blud is a portable build utility that uses Lua as its extension language. It was created to spark joy for the author in a way that Gnu make and cmake do not. This document assumes basic familiarity with make or any of its descendants. Famliarity with Lua is helpful, but familiarity with most any interpreted language that offers objects is probably sufficient, and free Lua tutorials abound.

Feldman's original make was a marvel of simplicity; a simple syntax for describing a dependency graph along with the actions required to rebuild missing dependents. One can implement a basic version of it in a single page of awk script.

The syntactic challenge for any modern make, however, is the build. Programmers often want to create slightly different builds of the same (or nearly the same) source files (a Windows build, a Linux build, a release build, a debug build, etc.). This is always possible in even the simplest make by just duplicating and then adjusting the original set of dependency graph rules but of course keeping nearly duplicate rule sets in sync is verbose and error-prone.

Feldman's make made liberal use of recursive macros and special character sequences to add features to the basic simple syntax. This trend often continued in make derivatives to the logical conclusion of drifting into incorporating a scripting language made cryptic by its roots. Incrementally developing a language starting from the primitives of make tends to result in a language that satisfies no one.

Instead of bolting on yet another peculiar language to the basic make syntax, blud is built on a simple, efficient, widely-known scripting language: Lua. The basic Lua syntax is extended to include the very simple basic make syntax for specifying dependencies.

Overview

Here is a simple blud script: 

program: program.c
    cc -o program program.c
As you can see it would also be valid input for make, or even most utilities derived from make.

Assuming this script was in a file called blud, executing the blud command would first rewrite this as Lua code into a file named blud.lua and then execute that code. Here roughly what blud.lua might look like in this case: 

-- DO NOT MODIFY: Generated by blud.
table.insert(OBJECTS["program"].PREREQUISITES, "program.c");
table.insert(OBJECTS["program"].ACTIONS, "cc -o program program.c");
BUILD.default = "program"
build()

Now suppose we want to have two different builds for our program: one will include debugging symbols and the other will not. 

debug : debug/program
release : release/program

debug/program : program.c
    cc -g -o debug/program program.c
release/program : program.c
    cc -o release/program program.c
Again, this is a valid specification for make or almost any make derivative. However, the rules have begun to become repetitive and have begun to obscure the fact that the debug and release builds are identical except for the location they store their object files and the compiler flags they use.

This is the fundamental syntactic challenge of any make utility: allowing arbitrary build customizations while succinctly capturing their similarities.

Note that make always contained a concept of implicit directory. There is no mention in the script of where program.c resides; it is assumed to reside in the current directory, that is, the directory the user was in when they ran blud. This idea leads to the first extension blud adds to basic make.
